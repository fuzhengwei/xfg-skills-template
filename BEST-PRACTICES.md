# 最佳实践（BEST-PRACTICES.md）

> 本文档总结了编写高质量 Agent Skills 的核心原则，基于 [agentskills.io 官方最佳实践](https://agentskills.io/skill-creation/best-practices) 整理。

---

# 技能编写最佳实践

## 1. 从真实专业知识出发

编写技能最常见的陷阱是：让 LLM 仅依靠其通用训练知识来生成技能，缺乏领域特定上下文。

有效的技能立足于**真实的专业知识**。关键是将领域特定上下文注入创建过程。

### 从实际操作任务中提取

与一个 Agent 完成一次真实任务，过程中提供上下文、纠正和偏好。然后将可复用模式提取到技能中。注意：

- **有效的步骤** — 导致成功的操作序列
- **你的纠正** — 你引导 Agent 采取不同方法的地方
- **输入/输出格式** — 数据进入和出来的样子
- **你提供的上下文** — Agent 原本不知道的项目特定事实、约定或约束

### 从现有项目产物中综合

当你有一批现有知识时，可以将其输入 LLM，让它综合成一个技能。基于团队实际事故报告和 runbook 综合的数据流水线技能，将优于基于通用"数据工程最佳实践"文章综合的技能。

好的源材料包括：
- 内部文档、runbook 和风格指南
- API 规范、模式和配置文件
- 代码审查评论和问题跟踪器（捕获反复出现的问题和审查者期望）
- 版本控制历史，特别是补丁和修复（通过实际变更揭示模式）
- 真实世界的故障案例及其解决方案

---

## 2. 通过真实执行来精炼

技能的初稿通常需要精炼。针对真实任务运行技能，然后将结果（全部结果，不仅仅是失败）反馈到创建过程中。

即使只有一次"执行→修订"的循环，也能明显提高质量。

> 💡 阅读 Agent 执行轨迹，而不仅仅是最终输出。如果 Agent 在无效步骤上浪费时间，常见原因包括：指令过于模糊（Agent 在找到有效方法之前尝试了多种方法）、指令不适用于当前任务（Agent 无论如何都遵循它们），或者给出了太多选项而没有明确的默认选项。

---

## 3. 明智地花费上下文（Token 预算）

技能激活后，其完整的 `SKILL.md` 正文会加载到 Agent 的上下文窗口中，与对话历史、系统上下文和其他活动技能一起竞争。技能中的每个 Token 都在争夺 Agent 的注意力。

### 添加 Agent 缺乏的内容，省略它已知的内容

专注于 Agent **没有**你的技能就不会知道的内容：项目特定约定、领域特定程序、不明显的边缘情况，以及要使用的特定工具或 API。

❌ 过于冗长 — Agent 已经知道 PDF 是什么：
```markdown
## 提取 PDF 文本
PDF（便携式文档格式）文件是一种常见文件格式，包含文本、图像和其他内容...
```

✅ 更好 — 直接跳到 Agent 独自不会知道的内容：
```markdown
## 提取 PDF 文本
使用 pdfplumber 进行文本提取。对于扫描文档，回退到 pdf2image + pytesseract。
```

### 设计连贯的单元

决定技能应该覆盖什么，就像决定一个函数应该做什么：你希望它封装一个连贯的工作单元，能与其他技能良好组合。

- 范围过窄的技能会迫使单个任务加载多个技能，带来开销和冲突指令的风险
- 范围过宽的技能难以精确激活
- 查询数据库并格式化结果的技能可能是一个连贯单元，而同时覆盖数据库管理的技能可能试图做太多事情

### 保持适度细节

过于全面的技能可能比它们帮助的更多 — Agent 难以提取相关内容，并可能追求由不适用于当前任务的指令触发的非生产性路径。简洁、逐步的指导和一个工作示例往往优于详尽的文档。

### 用渐进式披露构建大型技能

规范建议将 `SKILL.md` 保持在 500 行和 5000 Token 以内 — 仅包含 Agent 每次运行需要的核心指令。当技能确实需要更多内容时，将详细的参考材料移到 `references/` 或其他目录中的单独文件中。

关键是告诉 Agent *何时* 加载每个文件。"如果 API 返回非 200 状态码，请读取 `references/api-errors.md`" 比通用的"参见 references/ 获取详情"更有用。

## 4. 校准控制程度

并非技能的每个部分都需要相同水平的规范性。将指令的具体程度与任务的脆弱性相匹配。

### 将具体性与脆弱性相匹配

**给予 Agent 自由度** — 当多种方法都有效且任务容忍变化时。对于灵活的指令，解释 *为什么* 可能比刚性指令更有效。

```markdown
## 代码审查流程
1. 检查所有数据库查询是否存在 SQL 注入（使用参数化查询）
2. 验证每个端点的身份验证检查
3. 查找并发代码路径中的竞态条件
4. 确认错误消息不会泄露内部细节
```

**要规范化** — 当操作脆弱、一致性很重要，或者必须遵循特定序列时：

```markdown
## 数据库迁移
严格运行以下序列：
```bash
python scripts/migrate.py --verify --backup
```
不要修改命令或添加额外标志。
```

大多数技能是两者的混合。分别校准每个部分。

### 提供默认值，而非菜单

当多种工具或方法都可以工作时，选择一个默认值并简要提及替代方案，而不是将它们呈现为平等选项。

❌ 太多选项：
```markdown
你可以使用 pypdf、pdfplumber、PyMuPDF 或 pdf2image...
```

✅ 明确的默认值 + 逃生口：
```markdown
使用 pdfplumber 进行文本提取：
```python
import pdfplumber
```
对于需要 OCR 的扫描 PDF，请改用 pdf2image + pytesseract。
```

### 倾向于过程而非声明

技能应该教 Agent *如何处理* 一类问题，而不是为特定实例 *产生什么*。

❌ 特定答案 — 仅对完全相同的任务有用：
```markdown
将 `orders` 表与 `customers` 在 `customer_id` 上连接，过滤 `region = 'EMEA'`，并对 `amount` 列求和。
```

✅ 可复用方法 — 适用于任何分析查询：
```markdown
1. 从 `references/schema.yaml` 读取模式以找到相关表
2. 使用 `_id` 外键约定连接表
3. 将用户请求中的任何过滤器应用为 WHERE 子句
4. 根据需要聚合数字列并格式化为 Markdown 表格
```

## 5. 有效指令的模式

以下是在构建技能内容时可复用的技术。不是每个技能都需要所有这些 — 使用适合你任务的那些。

### Gotchas 部分

许多技能中价值最高的内容是一个 gotchas 列表 — 违背合理假设的环境特定事实。

```markdown
## Gotchas
- `users` 表使用软删除。查询必须包含 `WHERE deleted_at IS NULL`，否则结果将包含已停用的帐户。
- 用户 ID 在数据库中为 `user_id`，在身份验证服务中为 `uid`，在计费 API 中为 `accountId`。三者都引用相同的值。
- `/health` 端点只要 Web 服务器正在运行就返回 200，即使数据库连接已断开。使用 `/ready` 检查完整服务健康状态。
```

> 💡 当 Agent 犯了你需要纠正的错误时，将纠正措施添加到 gotchas 部分。这是迭代改进技能的最直接方式之一。

### 输出格式模板

当你需要 Agent 以特定格式产生输出时，提供模板。这比用散文描述格式更可靠。

```markdown
## 报告结构
使用此模板，根据具体分析调整各部分：

```markdown
# [分析标题]

## 执行摘要
[关键发现的一段概述]

## 关键发现
- 发现 1（附支持数据）
- 发现 2（附支持数据）

## 建议
1. 具体可执行的建议
2. 具体可执行的建议
```
```

### 多步骤工作流检查清单

显式检查清单帮助 Agent 跟踪进度并避免跳过步骤，特别是当步骤有依赖关系或验证门时。

```markdown
## 表单处理工作流
进度：
- [ ] 步骤 1：分析表单（运行 `scripts/analyze_form.py`）
- [ ] 步骤 2：创建字段映射（编辑 `fields.json`）
- [ ] 步骤 3：验证映射（运行 `scripts/validate_fields.py`）
- [ ] 步骤 4：填写表单（运行 `scripts/fill_form.py`）
- [ ] 步骤 5：验证输出（运行 `scripts/verify_output.py`）
```

### 验证循环

指示 Agent 在继续之前验证自己的工作。模式是：完成工作、运行验证器（脚本、参考检查清单或自检）、修复任何问题，然后重复直到验证通过。

### 计划-验证-执行

对于批量或破坏性操作，让 Agent 以结构化格式创建中间计划，根据真相源验证它，然后才执行。

## 6. 为 Agent 使用设计脚本

当 Agent 运行你的脚本时，它读取 stdout 和 stderr 来决定下一步做什么。几个设计选择使脚本对 Agent 来说显著更容易使用：

### 避免交互式提示（硬性要求）

Agent 在非交互式 shell 中运行 — 它们无法响应 TTY 提示、密码对话框或确认菜单。

✅ 所有输入通过命令行标志、环境变量或 stdin 接受

### 用 `--help` 记录用法

`--help` 输出是 Agent 学习脚本接口的主要方式。包含简要描述、可用标志和使用示例。

### 编写有用的错误消息

当 Agent 收到错误时，消息直接塑造其下一次尝试。不透明的 "Error: invalid input" 浪费一个回合。

✅ 说明出了什么问题、期望什么以及要尝试什么

### 使用结构化输出

优先使用结构化格式 — JSON、CSV、TSV — 而非自由格式文本。

💡 **将数据与诊断分开**：将结构化数据发送到 stdout，将进度消息、警告和其他诊断发送到 stderr。
